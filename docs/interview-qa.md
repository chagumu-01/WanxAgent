# WanxAgent 项目面试问答

> 基于 WanxAgent 智能对话系统项目的标准技术面试问答，涵盖系统架构、多 Agent 协作、RAG 知识库、MCP 协议、工程化等核心话题。

---

## 第一部分：项目整体架构

### Q1：用户输入一个问题后，系统从收到请求到返回答案，整个链路是怎么走的？

**答：**

简单任务走 RAG 直答链路：

```
 用户提问 → 向量库粗召回 → Reranker 重排序 → 上下文拼接 Prompt → LLM 生成结果
```

复杂任务走多 Agent 编排链路：

```
用户提问 → 意图分类(LLM) → 判定高复杂度 
→ Plan & Solve 规划图谱 → 任务分配 → Agent 并行执行 → 结果汇总 → 返回
```

两条链路由意图分类层统一分流，确保不同复杂度任务走不同执行路径。

---

### Q2：系统怎么判断一个问题应该走简单 RAG 还是走多 Agent 拆解编排？

**答：**

通过一次轻量级 LLM 调用做意图分类，分析任务的复杂度特征，而非简单规则匹配。判定为"难"之后，才进入 Plan & Solve 规划阶段。

最终演进方向是让意图分类输出多维度结构化的复杂度分析：

| 维度 | 含义 |
|------|------|
| 是否需要外部工具 | 问题涉及查数据、搜网页、调 API 吗？ |
| 子任务是否可独立执行 | 问题能拆成几个互不依赖的子问题吗？ |
| 推理链路深度 | 需要多步推理还是直接回答？ |
| 是否依赖多源信息 | 答案需要从多个不同知识源拼凑吗？ |

## 第二部分：多 Agent 协作与任务编排

### Q3：多 Agent 的分阶段是怎么设计的？怎么流转？

**答：**

六个阶段形成完整闭环：

| 阶段 | 做什么 | 负责组件 |
|------|--------|----------|
| 1. 意图分类 | 调 LLM 判定任务复杂度，决定路由 | 路由层 |
| 2. 任务发现 | 解析用户意图，识别所需知识和工具能力 | Planner Agent |
| 3. Plan 规划 | 动态生成 DAG 执行图谱，拆分出子任务及依赖关系 | Planner Agent |
| 4. 任务分配 | 子任务匹配最合适 Agent，注入对应工具集 | Harness |
| 5. 并行执行 | 无依赖子任务并发执行，Redis 同步进度 | Worker Agent |
| 6. 结果汇总 | 汇总所有子任务结果，拼装最终回复 | Aggregator Agent |

流转机制：Planner 发布规划 → Worker 订阅各自子任务 → 完成后发布完成事件 → Aggregator 订阅完成后触发汇总。

---

### Q4：用一个简单例子走一遍完整流程

**场景：** 用户输入"帮我分析新能源汽车行业发展趋势并写一份报告"

1. **意图分类**：复杂度高、多源信息、需多步推理 → 判定为复杂任务，走多 Agent 编排
2. **任务发现**：Planner 识别需搜索行业新闻、收集政策文件、查询市场数据、分析竞品、撰写报告
3. **Plan 规划**：生成 DAG 图谱：
   ```
   report_root
       ├── search_news（搜索 Agent）
       ├── search_policy（搜索 Agent）← 与上面并行
       ├── query_market（数据 Agent）← 与上面并行
       └── write_report（写作 Agent）← 等上面三个都完成
   ```
4. **任务分配**：搜索 Agent 挂网页搜索 + RAG；数据 Agent 挂数据库查询；写作 Agent 挂文档生成
5. **并行执行**：三个搜索/数据任务同时跑，Redis 同步进度
6. **结果汇总**：三路结果返回后，写作 Agent 拼装综合报告，流式推送

### Q5：Harness 都做了什么？

**答：**

Harness 是位于 LangChain Agent 之下的执行底座层，五个核心职责：

1. **Agent 生命周期管理**：独立容器化运行，控制创建、启动、暂停、销毁，配合 Redis Pub/Sub 做跨 Agent 上下文切换和状态同步
2. **工具注册与发现**：MCP 服务器上传后自动创建隔离客户端、拉取工具列表、注册到全局表，广播 `tools/list_changed` 通知所有 Agent 刷新
3. **安全调用网关**：所有工具调用经过 Host 中转——权限校验、参数清洗、审计记录、熔断控制，大模型永不能直连底层资源
4. **任务编排引擎**：基于 Plan & Solve 策略动态生成 DAG，控制子任务的串并行依赖关系和执行顺序
5. **流式输出管理**：重写 StreamingResponse 监听 disconnect 信号，用户中断时取消 Redis 任务标记位、停止模型生成

## 第三部分：RAG 知识库检索

### Q6：RAG 系统的记忆怎么设计？提取和召回分别怎么做？

**答：**

记忆分三层：

**短期记忆（对话上下文）**
- 存储：当前会话对话历史存在 Redis，按 `dialog_id` 隔离
- 提取：每轮对话自动追加，超 token 窗口时滑动截断
- 召回：直接拼接进下一轮 prompt

**长期记忆（会话历史）**
- 存储：持久化到数据库，按 `user_id + agent_name` 索引
- 提取：跨会话可查，历史对话完整保留
- 召回：用户切换历史对话时从数据库按 `dialog_id` 加载

**知识库记忆（RAG 知识）**
- 提取：PDF/Word/Markdown 解析 → 语义分块 → Embedding 向量化 → 存入 ChromaDB/Milvus
- 召回：混合检索（关键词 BM25 + 语义相似度）粗召回 → Reranker 重排序 → Top-K 注入 Prompt

## 第四部分：MCP 协议与工具调用

### Q7：MCP 服务器动态加载后，工具注册、发现、调用三个环节怎么串起来？

**答：**

全链路：**上传配置 → 新建客户端拉取工具 → 注册表注册并广播事件 → 所有 Agent 刷新工具集识别新工具 → 用户触发调用 → Host 安全中转路由至新 MCP 服务执行。**

关键设计：
- 每上传一份新 MCP Server 配置，Host 新建独立隔离的 Client
- 内置动态工具变更通知 `notifications/tools/list_changed`
- 实现运行时热更新，无需重启 Agent

---

### Q8：多个 MCP 服务器提供同名工具怎么办？安全网关怎么做的？

**答：**

**命名空间前缀**：自动为每个 MCP 服务的工具加前缀暴露给 Function Calling。例如：
- 内网文档服务 `search` → `doc_server__search`
- 外网搜索 `search` → `web_server__search`

同时在工具描述中补充数据源用途，LLM 能根据上下文需求直接选择对应工具，不存在识别混淆。

**安全网关**：Host 作为唯一中转网关，从配置接入、工具注册、调用转发、结果清洗到审计熔断全链路管控。核心原则是最小权限、默认拒绝——大模型与底层资源完全隔离，密钥永不过墙。

## 第五部分：工程化与对比分析

### Q9：Agent 和 Web Search 有本质区别吗？

**答：**

不能替代，四点本质差异：

| 维度 | Web Search | Agent |
|------|-----------|-------|
| **推理链路** | 搜一下→返回结果 | 理解问题→规划步骤→调用多工具→综合生成 |
| **私有知识** | 只搜公开网页 | 接入企业内部文档、数据库、私有 API |
| **执行动作** | 返回链接列表 | 发邮件、写文件、操作数据库，直接产出结果 |
| **多轮上下文** | 每次零上下文独立查询 | 记住对话历史，持续上下文中协作 |

概括：Web Search 是查字典，Agent 是请一个能查字典、会分析、能动笔的助手。

### Q10：用户凭什么用你这个项目，不用 Claude Code？

**答：**

定位不同。Claude Code 是**开发者的个人编程助手**，WanxAgent 是**企业级可定制的多 Agent 协作平台**。

| 维度 | Claude Code | WanxAgent |
|------|-------------|-----------|
| 部署方式 | Anthropic 云端 | 私有化部署，数据不出域 |
| 模型绑定 | 绑死 Claude | 多模型路由，自由切换 |
| 知识接入 | 项目文件 | 企业知识库 + 向量数据库 |
| 工具扩展 | 预设固定 | MCP 协议动态加载 |
| 协作模式 | 单人单 Agent | 多 Agent 分工并行 |
| 治理能力 | 无 | 审计日志、Token 看板、熔断 |
| 适用场景 | 个人写代码 | 企业智能客服、内部知识问答、业务自动化 |

## 第六部分：项目开发实践

### Q11：为什么想找 Agent 应用开发这个工作？

**答：**

LLM 本身是"大脑"但没有手脚，Agent 就是装上手脚——能调工具、能查知识库、能多步推理。从"会聊天"到"能干活"是质变。判断未来决定 AI 应用价值的关键不是模型，而是 Agent 的工程化能力。WanxAgent 验证了完整链路后，确认了这个方向是自己想深耕的，想在基础设施成形期积累从 0 到 1 的经验。
