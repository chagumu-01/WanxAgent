## 如何向我提问，来高效阅读 WanxAgent（提问模板）

这份文档的目标是：你不需要自己“翻仓库”，只要按模板提问，我就能按 **模块 → 入口 → 数据流 → 关键实现 → 可改点** 的方式把项目讲清楚，并能直接定位到文件/函数。

---

## 0) 你提问时，我最需要的 3 个信息

- **你的目标**：你想“理解”、还是“复述面试”、还是“准备改造/二开/排错”？
- **你的范围**：只看某个模块（如 RAG/Tool/MCP/对话流式/鉴权）还是全链路？
- **你的约束**：是否要离线、是否能访问外网、是否有 Milvus/ES/Redis/MySQL、是否要私有化部署？

你可以直接复制这一段填空：

```text
目标：{理解/面试复述/二开/排错}
范围：{全链路/只看RAG/只看MCP/只看对话接口/...}
约束：{离线/可外网}；{有/没有} Redis/MySQL/ES/Milvus；部署方式：{本地/Docker/服务器}
我当前卡点：{一句话}
```

## 1) “从 0 读懂项目”的推荐提问路线（最省时间）

按这 7 步问，我会给你一份“读代码路线图 + 关键文件清单 + 数据流图（文字版）”：

1. **项目入口/启动方式**
2. **后端路由总览：有哪些 API**
3. **一次对话请求的完整链路（从前端到落库）**
4. **Agent 如何初始化、如何选择模型、如何调用工具**
5. **RAG：上传文件 → 解析/分块 → 入库 → 检索 → rerank**
6. **MCP：服务器如何接入、工具如何转换与执行**
7. **运维与质量：鉴权/中间件/健康检查/配置管理**

你可以直接这样问：

```text
请给我一份 WanxAgent 的“阅读路线图”：按启动入口→后端路由→对话链路→Agent→RAG→MCP→配置与部署的顺序，
每一步列出关键文件、关键函数、数据流（输入/输出），并告诉我每一步我应该重点看什么。
```

## 2) 模块化提问模板（我会直接定位到文件/函数）

### 2.1 对话与流式输出（SSE）

```text
请你带我读“对话流式接口”的实现：从路由入口开始，解释每一步做了什么、为什么这么做。
重点回答：
1) SSE 的事件格式是什么？
2) 断开连接时如何停止任务/释放资源？
3) 历史记录/记忆是怎么注入 prompt 的？
4) 最终哪些数据会落库（文本/事件/统计）？
```

### 2.2 RAG（知识库）

```text
请你带我读 RAG 的完整链路：知识库创建→文件上传→解析/分块→向量化→(可选)ES索引→检索→重排序→返回。
同时告诉我：
- 混合检索是怎么合并/去重的？
- rerank 的输入输出是什么？
- top_k/min_score 等参数在哪配置，如何调参？
```

### 2.3 Tool Calling（OpenAPI 工具）

```text
请你解释“用户上传 OpenAPI Schema → 变成可调用工具”的实现。
我想知道：schema 怎么校验、工具怎么存、运行时怎么被 Agent 使用、失败时如何返回错误。
```

### 2.4 MCP（外部能力扩展）

```text
请你解释 MCP 集成：如何连接 MCP server、如何列出工具、如何把 MCP 工具转换为 LangChain tool，
以及调用时返回的 Text/非文本内容如何处理。
最后给我一个“新增一个 MCP server”的最小改动清单。
```

### 2.5 记忆系统（Memory）

```text
请你解释 enable_memory 打开后：对话时如何检索记忆、如何写入记忆、如何做事实抽取/更新/删除，
以及如何用 user_id/agent_id/run_id 做隔离。
```

### 2.6 鉴权/中间件/初始化

```text
请你解释后端启动时的生命周期初始化做了什么（数据库/默认Agent/MCP等），JWT 鉴权如何配置，
TraceID/白名单中间件的作用是什么，/health 用于什么场景。
```

## 3) 排错类问题的“高质量提问法”（我能直接定位问题）

把下面信息一次性贴出来，我就能复现或直接定位：

- **你做了什么**：哪条命令/哪个页面/哪个接口
- **你期望什么** vs **实际发生什么**
- **报错原文**（不要截图，复制文本更好）
- **相关请求参数**（如 dialog_id、knowledge_id、tool_id、file_url）
- **环境**：Windows/Linux、Python/Node 版本、是否使用脚本 `scripts/start.py`

模板：

```text
我在 {环境} 用 {命令/页面} 触发了 {功能}。
期望：{...}
实际：{...}
报错日志：{粘贴原文}
相关参数：{dialog_id=..., knowledge_id=..., ...}
我怀疑的模块：{SSE/RAG/MCP/DB/前端...}
请你按“最可能原因→验证方法→修复方案→需要改的文件”来回答。
```

## 4) 想“像简历一样讲出来”的提问法

你可以让我帮你把某个模块整理成“面试话术 + 技术点 + 难点/取舍 + 可量化结果”的结构：

```text
请把 {模块名：RAG/流式对话/MCP/记忆系统/工具调用} 总结成一段 30 秒自我介绍 + 3 个可追问点。
每个追问点给出：涉及的关键文件、你建议我怎么讲“为什么这么设计”、以及一个可优化方向。
```

## 5) 我输出的默认形式（你可以要求我换）

如果你不特别说明，我通常会输出：

- **关键文件清单**（按阅读顺序）
- **一次请求的数据流**（输入 → 处理 → 输出 → 落库）
- **关键函数/类**（讲它们如何协作）
- **你能怎么改**（最小改动方案 + 风险点）

如果你更想要“代码级讲解”，你可以额外加一句：

```text
请你在回答里用“文件路径 + 函数名 + 关键代码片段”来引用，不要泛泛而谈。
```

## 6) 本次对话的提问偏好总结

你这次的需求可以总结为两类输出，后续你只要把下面要求贴给我，我就会按同样风格产出。

### 6.1 写简历式项目亮点时

- 要点数量要少 3 到 6 条即可  
- 每条是一句话 先用一个词或短语做概括 再用一句话把功能讲清楚  
- 不要出现双引号 括号 代码函数名 文件路径 箭头符号 这类容易显得像 AI 的写法  
- 句式偏工程与价值导向 说明做了什么 解决了什么问题 带来什么收益  

你可以直接这样问我

```text
请把项目亮点按简历写法输出 4 到 6 条 每条前面加一个概括词 每条一句话
不要出现双引号 括号 代码函数名 文件路径 箭头符号
```

### 6.2 读项目文档与讲解时

- 先给路线图 再按模块逐步深入 需要时再引用关键代码  
- 如果你只想快速过一遍 就要求我只讲整体链路与关键模块  
- 如果你要面试复述 就要求我把每个模块压成一句话加价值点  

你可以直接这样问我

```text
先给我读代码路线图 只列模块与我应该看什么 不要展开
等我确认后 再按我指定模块深入讲实现细节
```
